# Index
인덱스란 추가적인 작업들과 별도의 저장 공간을 이용하여 테이블의 검색 속도를 향상시키기 위한 자료 구조입니다.

- 인덱스를 구현하기 위해서는 `Hash Table`과 `B+ Tree` 등을 사용할 수 있습니다.
- `UPDATE`나 `DELETE` 쿼리는 `SELECT`를 거쳐야만 수행이 가능하기 때문에 인덱스를 적용하면 `UPDATE`와 `DELETE`의 성능도 함께 향상됩니다.

`DBMS`는 인덱스를 항상 최신으로 정렬된 상태여야만 성능을 향상할 수 있습니다. 이로 인해 `INSERT`, `DELETE`, `UPDATE`가 수행될 때 인덱스에 대한 오버헤드가 다음과 같이 발생합니다.

- `INSERT`: 새로 추가된 데이터에 인덱스를 적용하는 작업을 진행합니다.
- `DELETE`: 삭제할 데이터를 인덱스에서 제외시키는 작업을 진행합니다.
- `UPDATE`: 수정할 데이터를 인덱스에서 제외하고, 수정된 데이터를 인덱스에 추가하는 작업을 진행합니다.
## 인덱스 구조
인덱스의 구조는 `key`와 `value`로 이루어져있습니다. 여기서 `key`는 검색에 사용되는 컬럼이나 컬럼 조합을 의미하고, `value`는 해당 레코드에 가리키는 포인터를 말합니다.
## 장점
- 데이터를 조회하는 속도와 그에 따른 수정, 삭제 등에 성능을 향상 시킬 수 있습니다.
- 시스템에 부하를 줄일 수 있습니다.
## 단점
- 인덱스를 관리하기 위해서는 사용 DB의 약 10% 정도의 공간이 별도로 필요합니다.
- 인덱스를 관리하기 위해서 추가 작업이 필요합니다.
- 인덱스를 올바르지 않은 방법으로 사용할 시 오히려 성능이 저하될 수 있습니다.
## 인덱스를 사용하면 좋은 경우
- 테이블의 규모가 작지 않은 경우
- `INSERT`, `UPDATE`, `DELETE`가 자주 발생하지 않는 컬럼
- `JOIN`, `WHERE`, `ORDER BY`가 자주 사용된는 컬럼
- 데이터의 중복도가 낮은 컬럼
